## 2. 구현한 요청 처리 흐름

```
[관리 흐름]  회원가입/로그인 → JWT 발급 → (JwtAuthFilter) → 키 발급/목록/폐기
[프록시 흐름] POST /v1/chat/completions + tlm_ 키
  → ApiKeyAuthFilter : SHA-256 해시로 ACTIVE 키 조회, 사용자 식별
  → ProxyService     : 요청 본문에서 model 추출
  → ProviderRouter   : gpt-*/o* → OpenAI, claude* → Anthropic
  → OpenAiClient(통과) / AnthropicClient(형식 변환 어댑터)
  → 응답 반환 (프로바이더 오류는 502 PROVIDER_ERROR로 변환)
```

## 3. 핵심 설계 결정과 근거

| 결정 | 근거 |
|---|---|
| 비밀번호 bcrypt, API 키 SHA-256 | 입력 엔트로피·검증 빈도 기준. 비밀번호=저엔트로피·저빈도 → 느린 해시로 무차별 대입 방어. API 키=256비트 랜덤·매 요청 검증 → 빠른 해시 + 결정적이라 유니크 인덱스 단건 조회 |
| 키 원문 미저장 (해시만) | DB 유출 시에도 키 원문 보호. 원문은 발급 응답에서 1회만 노출 |
| 로그인 실패 사유 단일 메시지 | "이메일 없음"과 "비밀번호 틀림"을 구분하면 계정 열거 공격 가능 |
| 회원가입 @Transactional | User·UsageQuota($5) 이중 저장의 원자성 — 부분 성공 시 "쿼터 없는 유저" 방지 |
| JWT (세션 대신) | 무상태 → 수평 확장 시 세션 공유 불필요, API 클라이언트에 자연스러움. 무효화 불가 단점은 짧은 만료(1h)로 완화 |
| OpenAI 호환 형식 + 어댑터 패턴 | 기존 도구가 base URL만 바꿔 그대로 사용. 프로바이더 형식 차이는 AnthropicClient가 내부 흡수 |
| RestClient 타임아웃 (3s/60s) | 외부 API 지연 시 톰캣 스레드 풀 고갈 → 서버 전체 마비 방지 |
| 인증 이원화 (JWT vs API 키) | 사람(관리 API)은 로그인 기반 JWT, 도구(프록시)는 발급 키 — 사용 주체가 다름 |

## 5. 트러블슈팅 기록

1. **JDK 21 vs toolchain 17** — 로컬 JDK 21뿐인데 toolchain 17 지정 → 21로 통일 (Boot 3.3 공식 지원 LTS)
2. **`jakarta` vs `spring`의 @Transactional** — 같은 이름 어노테이션 임포트 혼동. 스프링 것(readOnly 등 옵션 보유)으로 통일
3. **필터의 예외는 @RestControllerAdvice가 못 잡음** — 필터는 컨트롤러 바깥 단계라 JSON 에러 응답을 필터에서 직접 작성
4. **테스트 시 PowerShell → curl.exe JSON 이스케이프 깨짐** — 서버 버그로 오인할 뻔. Invoke-RestMethod로 전환해 해결 (서버 로그의 JSON parse error로 원인 확인)

## 6. 학습 기록 — 구현하며 스스로 묻고 답한 질문들

> 기능마다 "왜 이렇게 하는가"를 먼저 정리하고 코드를 작성했다. 면접 예상 질문 형식으로 기록한다.

**Q. 비밀번호는 bcrypt로 해싱하면서 API 키는 왜 SHA-256인가?**
해싱 방식은 입력의 엔트로피와 검증 빈도로 결정했다. 비밀번호는 사람이 만들어 추측 가능(저엔트로피)하고 로그인 시에만 검증되므로, 일부러 느린 bcrypt로 무차별 대입을 막는다. API 키는 서버가 만든 256비트 랜덤이라 대입 공격 자체가 불가능하고, 매 요청마다 검증되므로 빠른 SHA-256을 쓴다. 부가 이점: SHA-256은 결정적(같은 입력→같은 출력)이라 해시 컬럼에 유니크 인덱스를 걸어 단건 조회가 가능하다. bcrypt는 매번 salt가 달라 이 조회가 불가능하다.

**Q. 세션 대신 JWT를 선택한 이유는? 단점은 없나?**
JWT는 토큰 안에 신원과 서명이 자체 포함된 무상태 방식이라, 서버가 세션 저장소를 조회·공유할 필요가 없어 수평 확장에 유리하고, SDK/CLI 같은 API 클라이언트의 Authorization 헤더 방식에 자연스럽다. 단점은 서버가 상태를 안 들고 있어 만료 전 강제 무효화가 불가능하다는 것 — 짧은 만료(1시간)로 완화했다. 참고로 JWT 페이로드는 암호화가 아니라 서명이므로(Base64로 누구나 열람 가능) 민감정보를 넣지 않는다.

**Q. 회원가입 메서드에 @Transactional을 붙인 이유는?**
User 저장과 UsageQuota($5) 저장, 쓰기가 2번이다. 중간에 실패하면 "쿼터 없는 유저"가 생겨 이후 비용 한도 체크가 깨진다. @Transactional로 두 저장을 묶어 전부 성공 아니면 전부 롤백(원자성)을 보장했다. 스프링이 AOP로 메서드를 감싸 BEGIN/COMMIT/ROLLBACK을 대신 처리한다.

**Q. 로그인 실패 시 "이메일 없음"과 "비밀번호 틀림"을 왜 구분하지 않나?**
메시지를 구분하면 공격자가 임의 이메일을 넣어보며 가입 여부를 알아낼 수 있다(계정 열거 공격). 두 실패를 같은 메시지("이메일 또는 비밀번호가 올바르지 않습니다")로 뭉개서 존재 여부를 숨긴다.

**Q. 키 폐기(revoke)에서 save() 호출 없이 어떻게 DB가 갱신되나?**
JPA 변경 감지(dirty checking). @Transactional 안에서 조회한 엔티티는 영속성 컨텍스트가 스냅샷을 들고 있다가, 값이 바뀌면 커밋 시점에 UPDATE를 자동 실행한다.

**Q. 인증 필터에서 발생한 401은 왜 GlobalExceptionHandler로 처리하지 않나?**
서블릿 필터는 DispatcherServlet(컨트롤러) 앞 단계에서 실행되는데, @RestControllerAdvice는 컨트롤러 영역의 예외만 잡는다. 그래서 필터에서는 JSON 에러 응답을 직접 작성한다.
